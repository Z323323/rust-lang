The `join!` macro is convenient when you know at compile time exactly how many futures you want to await concurrently, but real applications often work with a *dynamic* number of futures — a list of URLs to fetch, a set of pending timers, or an unknown-length batch of tasks. Rust provides combinators like `join_all`, `select!`, and pinned trait objects to handle these cases ergonomically.

## Awaiting a Dynamic List of Futures with `join_all`

When you have a `Vec` of futures that all return the same type, `futures::future::join_all` (re-exported by many runtimes) awaits all of them concurrently and returns a `Vec` of their outputs in the original order.

```rust
use std::time::Duration;

async fn fetch(id: u32) -> String {
    trpl::sleep(Duration::from_millis(id as u64 * 100)).await;
    format!("result-{id}")
}

fn main() {
    trpl::run(async {
        let futures: Vec<_> = (1..=5).map(fetch).collect();
        let results = trpl::join_all(futures).await;
        for r in results {
            println!("{r}");
        }
    });
}
```

All five futures run concurrently; the total wall time is approximately 500 ms (the longest single delay), not 1500 ms. Results are returned in the *original* order regardless of which future finishes first.

## Futures That Return Different Types

`join_all` requires all futures to have the same `Output` type. When futures return different types, you must unify them — typically by boxing them as `Pin<Box<dyn Future<Output = SomeCommonType>>>`:

```rust
use std::future::Future;
use std::pin::Pin;

type BoxedFuture<'a> = Pin<Box<dyn Future<Output = String> + 'a>>;

async fn get_name() -> String { String::from("Alice") }
async fn get_age() -> String { 30u32.to_string() }

fn main() {
    trpl::run(async {
        let futures: Vec<BoxedFuture<'_>> = vec![
            Box::pin(get_name()),
            Box::pin(get_age()),
        ];
        let results = trpl::join_all(futures).await;
        println!("{:?}", results); // ["Alice", "30"]
    });
}
```

`Box::pin(future)` allocates the future on the heap and pins it in place. The `Pin` wrapper prevents the future from being moved in memory — a requirement for futures that contain self-referential state (explained further in the Async Traits chapter).

## Why Pinning Matters

Async state machines generated by `async`/`.await` can be *self-referential*: one field of the state machine holds a reference to another field. If such a struct were moved in memory, the reference would become invalid. `Pin<P>` is a wrapper that says: *the value behind this pointer will not be moved*. The compiler enforces this.

```rust
use std::pin::Pin;

// `pin_mut!` pins a future on the *stack* (from the `pin-utils` crate or std::pin::pin!).
fn main() {
    trpl::run(async {
        let mut future = std::pin::pin!(async {
            trpl::sleep(Duration::from_millis(100)).await;
            42u32
        });
        // Now we can poll `future` directly if needed.
        let value = future.await;
        println!("{value}");
    });
}
```

In practice, you mostly encounter pinning when storing futures in collections or writing manual `Future` implementations. The `Box::pin` pattern is the easiest escape hatch.

## Racing with `select!`

While `join_all` waits for *all* futures, `select!` (from the `futures` crate or Tokio) resolves as soon as the *first* future completes, and drops the rest:

```rust
use std::time::Duration;

fn main() {
    trpl::run(async {
        let slow = async {
            trpl::sleep(Duration::from_secs(2)).await;
            "slow result"
        };
        let fast = async {
            trpl::sleep(Duration::from_millis(50)).await;
            "fast result"
        };

        // Syntax varies by crate; here we use trpl::race for simplicity.
        let winner = trpl::race(slow, fast).await;
        println!("Winner: {winner}");
        // Prints "Winner: fast result" after ~50 ms.
    });
}
```

Tokio's `select!` macro supports multiple arms and pattern matching on results:

```rust
// Tokio-specific example (illustrative):
tokio::select! {
    result = database_query() => {
        println!("DB returned: {result:?}");
    }
    _ = tokio::time::sleep(Duration::from_secs(5)) => {
        println!("Database timed out");
    }
}
```

## Adding Timeouts

A common pattern combines `select!` (or `race`) with a sleep future to implement operation timeouts cleanly:

```rust
async fn with_timeout<F, T>(duration: Duration, future: F) -> Option<T>
where
    F: Future<Output = T>,
{
    trpl::race(
        async { Some(future.await) },
        async {
            trpl::sleep(duration).await;
            None
        },
    )
    .await
}

fn main() {
    trpl::run(async {
        let result = with_timeout(
            Duration::from_millis(200),
            simulate_slow_db_query(),
        )
        .await;

        match result {
            Some(data) => println!("Got data: {data}"),
            None => println!("Query timed out"),
        }
    });
}

async fn simulate_slow_db_query() -> String {
    trpl::sleep(Duration::from_secs(1)).await;
    String::from("query result")
}
```

## Handling Cancellation

When a future is dropped (for example, the losing branch of `select!`), Rust's normal drop mechanics run. Any state the future owns is cleaned up — file handles are closed, locks are released. However, *partially executed* async work is simply abandoned at the most recent `.await` point. This means:

- You should not hold locks across `.await` points in a `select!` branch (the lock may be held if the branch is cancelled).
- Avoid performing non-idempotent side effects before a yield point if the future might be cancelled.

```rust
async fn safe_update(counter: Arc<Mutex<u32>>) {
    // Acquire lock, do work, release — all without yielding.
    // This is safe to cancel: the lock is not held across an await.
    {
        let mut val = counter.lock().unwrap();
        *val += 1;
    } // Lock released here.
    trpl::sleep(Duration::from_millis(10)).await; // Cancellation point.
}
```

## Summary of Composition Patterns

| Pattern | Use when… |
|---|---|
| `join!` | Fixed small number of futures, all needed |
| `join_all(vec)` | Dynamic number of same-typed futures |
| `Box::pin` + `join_all` | Dynamic list of different-typed futures |
| `race` / `select!` | Need the first to complete; rest can be dropped |
| Timeout + `race` | Any operation that should have a deadline |
