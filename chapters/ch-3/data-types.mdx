>Every value in Rust is of a certain <u>data type</u>, which tells Rust what kind of data is being specified so that it knows how to work with that data. We’ll look at two data type subsets: scalar and compound.

>Keep in mind that Rust is a statically typed language, which means that <u>it must know the types of all variables at compile time</u>. The compiler can usually infer what type we want to use based on the value and how we use it. In cases where multiple types are possible, we must add a type annotation, as shown in Code 3.3.1 below. If we omit the `: u32` annotation, Rust will display the compilation error seen in C-Output 3.3.1.

<CodeGroup>
```rust Code 3.3.1
let guess: u32 = "42".parse().expect("Not a number!");
```

## Scalar Types

>A scalar type represents a <u>single value</u>. Rust has four primary scalar types: integers, floating-point numbers, Booleans, and characters.

### Integer Types

>An integer is a number without a fractional component.

| Length | Signed  | Unsigned |
| ---: | :---: |  :---  |
| 8-bit | `i8` | `u8` |
| 16-bit | `i16` | `u16` |
| 32-bit | `i32` | `u32` |
| 64-bit | `i64` | `u64` |
| 128-bit | `i128` | `u128` |
| Arch.-dependent | `isize` | `usize` |

>The `isize` and `usize` types depend on the architecture of the computer your program is running on: 64 bits if you’re on a 64-bit architecture and 32 bits if you’re on a 32-bit architecture.

>Each signed variant can store numbers from $-2^{n - 1}$ to $2^{n - 1} - 1$ inclusive, where $n$ is the number of bits that variant uses. So, an `i8` can store numbers from $-2^{7}$ to $2^{7 - 1}$, which equals $-128$ to $127$. Unsigned variants can store numbers from $0$ to $2^{n} - 1$, so a `u8` can store numbers from $0$ to $2^{8} - 1$, which equals $0$ to $255$.

Let $b_{j} \in \{ 0, 1 \}$ for $j \in \{ 0, \dots, n - 1 \}$ and $c \in \{ - 1, 0 \}$. Then we have

```

```

If you notice anything 'strange' in Code 3.3.2, it is likely a workaround to avoid <u>compiler errors related to integer overflow</u>. For example, <u>we cannot directly compute `2_i8.pow(7) - 1` because the compiler evaluates the exponentiation $2^7$ before subtracting $1$; the intermediate step overflows the maximum capacity of an `i8` variable (which is $2^{7} - 1$)</u>. The Rust compiler is incredibly strict and will reject the code if it detects a guaranteed overflow at compile time. <u>This strictness also explains why writing `-2_i8.pow(7)` fails to compile (notice that we instead wrote `(-2_i8).pow(7)`; there are parentheses around `-2_i8`!), even though the final mathematical result ($- 2^{7} = - 128$) is the exact minimum value that fits into an `i8`. The issue is operator precedence: the compiler attempts to evaluate `2_i8.pow(7)` first, triggering an overflow before the minus sign is ever applied! To fix this, we must wrap the negative number in parentheses, writing `(-2_i8).pow(7)`, ensuring the base is negative from the start</u> (as I did in Code 3.3.2 above).

<Info>
**Integer Overflow**

>Let’s say you have a variable of type `u8` that can hold values between 0 and 255. If you try to change the variable to a value outside that range, such as 256, <u>integer overflow</u> will occur, which can result in <u>one of two</u> behaviors.

>1. When you’re compiling in <u>debug mode</u>, Rust includes checks for integer overflow that cause your program to **panic** at runtime if this behavior occurs. Rust uses the term <u>panicking</u> when a program exits with an error.

>2. When you’re compiling in <u>release mode</u> with the `--release` flag, <u>Rust does not include checks for integer overflow that cause panics</u>. Instead, if overflow occurs, Rust performs <u>two’s complement wrapping</u>. In short, <u>values greater than the maximum value the type can hold “wrap around” to the minimum of the values the type can hold</u>. In the case of a `u8`, the value 256 becomes 0, the value 257 becomes 1, and so on. The program won’t panic, but the variable will have a value that probably isn’t what you were expecting it to have. <u>Relying on integer overflow’s wrapping behavior is considered an error.</u>

>To explicitly handle the possibility of overflow, you can use these families of methods provided by the standard library for primitive numeric types:

>- Wrap in all modes with the `wrapping_*` methods, such as `wrapping_add`.
>- Return the `None` value if there is overflow with the `checked_*` methods.
>- Return the value and a Boolean indicating whether there was overflow with the `overflowing_*` methods.
>- Saturate at the value’s minimum or maximum values with the `saturating_*` methods.
</Info>

### Floating-Point Types

>Rust also has two primitive types for floating-point numbers, which are numbers with decimal points. Rust’s floating-point types are `f32` and `f64`, which are 32 bits and 64 bits in size, respectively. The default type is `f64` because on modern CPUs, it’s roughly the same speed as `f32` but is capable of more precision. All floating-point types are signed.

```rust Code 3.3.4
fn main() {
    let x = 2.0; // f64

    let y: f32 = 3.0; // f32
}
```

### Numeric Operations

>Rust supports the basic mathematical operations you’d expect for all the number types: addition, subtraction, multiplication, division, and remainder.

>Integer division truncates toward zero to the nearest integer.<br />

```rust Code 3.3.5
fn main() {
    // ADDITION
    let sum = 5 + 10;

    // SUBTRACTION
    let difference = 95.5 - 4.3;

    // MULTIPLICATION
    let product = 4 * 30;

    // DIVISION
    let quotient = 56.7 / 32.2;
    let truncated = -5 / 3; // Results in -1.

    // REMAINDER
    let remainder = 43 % 5;

```

It is worth highlighting a subtle distinction regarding the mathematical <u>floor function</u>. While flooring produces the exact same result as truncation for positive numbers, it behaves quite differently with negative numbers. This difference is demonstrated in Code 3.3.6 below!

```rust Code 3.3.6
fn main() {
    let num = -1.6_f32;
    println!("Truncate: {}", num.trunc()); // 'Truncate: -1'
    println!("Floor: {}", num.floor()); // 'Floor: -2'
}
```

**Floor** ($\lfloor x \rfloor$): The greatest integer less than or equal to $x$.

### The Boolean Type

>As in most other programming languages, a Boolean type in Rust has two possible values: `true` and `false`. Booleans are <u>one byte</u> in size. The Boolean type in Rust is specified using `bool`.

```rust Code 3.3.6
fn main() {
    let t = true;

    let f: bool = false; // With explicit type-annotation.
}
```

>The main way to use Boolean values is through conditionals, such as an `if` expression.

### The Character Type

>Rust’s `char` type is the language’s most primitive alphabetic type. Here are some examples of declaring `char` values.

```rust Code 3.3.7
fn main() {
    let c = 'z';
    let z: char = 'ℤ'; // With explicit type-annotation.
    let heart_eyed_cat = '😻';
}
```

>Note that <u>we specify `char` literals with single quotation marks, as opposed to string literals, which use double quotation marks</u>. <u>Rust’s `char` type is 4 bytes in size and represents a Unicode scalar value, which means it can represent a lot more than just ASCII.</u> Accented letters; Chinese, Japanese, and Korean characters; emojis; and zero-width spaces are all valid `char` values in Rust. Unicode scalar values range from `U+0000` to `U+D7FF` and `U+E000` to `U+10FFFF` inclusive. However, a “character” isn’t really a concept in Unicode, so your human intuition for what a “character” is may not match up with what a `char` is in Rust.

As we'll discover later, assuming every character always takes up exactly 4 bytes in memory can lead to trouble. <u>While Rust's built-in `char` type is exactly 4 bytes, when characters are stored inside a `String` or a `&str`, they are UTF-8 encoded. Inside a string, a character can take up anywhere from 1 to 4 bytes depending on what it is (e.g., standard ASCII is 1 byte, while an emoji might be 4 bytes).</u> It can get complicated, but don't worry—later on, we will write some code that handles this UTF-8 complexity effortlessly!

## Compound Types

><u>Compound types can group multiple values into one type.</u> Rust has two primitive compound types: **tuples** and **arrays**.

Do you remember back in section 3.2 when I mentioned that it is actually possible to return an arbitrary number of values from an inner scope? Compound types are a prime—though not the only—example of how this is achieved.

### The Tuple Type

>A tuple is a general way of grouping together a number of values with a variety of types into one compound type. <u>Tuples have a fixed length</u>: Once declared, they cannot grow or shrink in size.

>We create a tuple by writing a comma-separated list of values inside parentheses. Each position in the tuple has a type, and <u>the types</u> of the different values in the tuple <u>don’t have to be the same</u>. 

```rust Code 3.3.8
fn main() {
    let tup: (i32, f64, u8) = (500, 6.4, 1);
}
```

>The variable `tup` binds to the entire tuple because a tuple is considered a single compound element. To get the individual values out of a tuple, we can use pattern matching to <u>destructure</u> a tuple value, like this:

```rust Code 3.3.9
fn main() {
    let tup = (500, 6.4, 1);

    let (x, y, z) = tup;

    println!("The value of y is: {y}");
}
```

>This program first creates a tuple and binds it to the variable `tup`. It then uses a pattern with `let` to take `tup` and turn it into three separate variables, `x`, `y`, and `z`. This is called destructuring because it breaks the single tuple into three parts. Finally, the program prints the value of `y`, which is `6.4`.

>We can also access a tuple element directly by using a period (`.`) followed by the index of the value we want to access. For example:

```rust Code 3.3.10
fn main() {
    let x: (i32, f64, u8) = (500, 6.4, 1);

    let five_hundred = x.0;

    let six_point_four = x.1;

    let one = x.2;
}
```

>This program creates the tuple `x` and then accesses each element of the tuple using their respective indices. As with most programming languages, the first index in a tuple is 0.

>The tuple without any values has a special name, **unit**. <u>This value and its corresponding type are both written `()` and represent an empty value or an empty return type. Expressions implicitly return the unit value if they don’t return any other value.</u>

### The Array Type

>Another way to have a collection of multiple values is with an array. Unlike a tuple, <u>every element of an array must have the same type</u>. <u>Arrays in Rust have a fixed length.</u>

>We write the values in an array as a comma-separated list inside square brackets.

```rust Code 3.3.11
fn main() {
    let a = [1, 2, 3, 4, 5];
}
```

><u>Arrays are useful when you want your data allocated on the **stack** rather than the **heap** or when you want to ensure that you always have a fixed number of elements.</u> An array isn’t as flexible as the **vector** type, though. <u>A vector is a similar collection type provided by the standard library that is allowed to grow or shrink in size because its contents live on the heap. If you’re unsure whether to use an array or a vector, chances are you should use a vector.</u> However, arrays are more useful when you know the number of elements will not need to change. 

>For example, if you were using the names of the month in a program, you would probably use an array rather than a vector because you know it will always contain 12 elements:

```rust Code 3.3.12
let months = ["January", "February", "March", "April", "May", "June", "July",
              "August", "September", "October", "November", "December"];

```

>You write an array’s type using square brackets with the type of each element, a semicolon, and then the number of elements in the array, like so:

```rust Code 3.3.13
let a: [i32; 5] = [1, 2, 3, 4, 5];
```

>Here, `i32` is the type of each element. After the semicolon, the number `5` indicates the array contains five elements.

>You can also initialize an array to contain the same value for each element by specifying the initial value, followed by a semicolon, and then the length of the array in square brackets, as shown here:

```rust Code 3.3.14
let a = [3; 5];
```

>The array named `a` will contain `5` elements that will all be set to the value `3` initially. This is the same as writing `let a = [3, 3, 3, 3, 3];` but in a more concise way.

### Array Element Access

>You can access elements of an array using indexing, like this:

```rust Code 3.3.15
fn main() {
    let a = [1, 2, 3, 4, 5];

    let first = a[0];
    let second = a[1];
}
```

>In this example, the variable named `first` will get the value `1` because that is the value at index `[0]` in the array. The variable named `second` will get the value `2` from index `[1]` in the array.

### Invalid Array Element Access

>Let’s see what happens if you try to access an element of an array that is <u>past the end of the array</u>. <u>Code 3.3.16 compiles successfully.</u> If you run Code 3.3.16 using `cargo run` and enter `0`, `1`, `2`, `3`, or `4`, the program will print out the corresponding value at that index in the array. If you instead enter a number past the end of the array, such as `10`, you’ll see E-Output 3.3.16.

<CodeGroup>
```rust Code 3.3.16
use std::io;

fn main() {
    let a = [1, 2, 3, 4, 5];

    println!("Please enter an array index.");

    let mut index = String::new();

    io::stdin()
        .read_line(&mut index)
        .expect("Failed to read line");

    let index: usize = index
        .trim()
        .parse()
        .expect("Index entered was not a number");

    let element = a[index];

    println!("The value of the element at index {index} is: {element}");
}
```
```text E-Output 3.3.16
thread 'main' panicked at src/main.rs:19:19:
index out of bounds: the len is 5 but the index is 10
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```
</CodeGroup>

This is one of the rare instances I warned you about, where E-Output X.Y.Z actually shows a runtime error!

>The program resulted in a runtime error at the point of using an invalid value in the indexing operation. The program exited with an error message and didn’t execute the final `println!` statement. When you attempt to access an element using indexing, Rust will check that the index you’ve specified is less than the array length. If the index is greater than or equal to the length, Rust will panic. This check has to happen at runtime, especially in this case, because the compiler can’t possibly know what value a user will enter when they run the code later.

>This is an example of Rust’s memory safety principles in action. In many low-level languages, this kind of check is not done, and when you provide an incorrect index, invalid memory can be accessed. Rust protects you against this kind of error by immediately exiting instead of allowing the memory access and continuing. Ch. 9 discusses more of Rust’s error handling and how you can write readable, safe code that neither panics nor allows invalid memory access.
