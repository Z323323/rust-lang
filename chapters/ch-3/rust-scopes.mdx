Before moving on to Rust’s scalar types, let's revisit the inner scope from the previous shadowing example. We will use it as a foundation to explore how scopes work in Rust in much greater depth.

You can already spot a key characteristic of scopes just by looking at that code: **the shadowed variable `x` created inside the `{ ... }` block literally ceases to exist once that scope ends** (which is why we printed it while still inside).

One subtle detail that might be easy to overlook in the previous section example is this: **we can print the value of `x` in the outer scope after exiting the inner scope simply because shadowing a variable inside an inner block is exactly like defining a completely new variable**. Consider the following example:

```rust
fn main() {
    let x = 5;

    let x = x + 1; // 6 = 5 + 1

    {
        let a = x * 2; // 12 = 6 * 2
        println!("The value of a in the inner scope is: {a}");
    } // The inner scope ends here, dropping the inner 'a' as it goes out of scope.

    println!("The value of x is: {x}"); // Here, the outer 'x' is bound to the value 6.
}
```

Executing this latest snippet with `cargo run` produces the following output, demonstrating that both code examples behave exactly the same way.

```text
   Compiling scopes v0.1.0 (/var/home/z561/rust_projects/scopes)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.34s
     Running `target/debug/scopes`
The value of a in the inner scope is: 12
The value of x is: 6
```

Now, however, let's observe what happens when we try to access the value of `a` outside the inner scope. For example, by writing:

```rust
fn main() {
    // --snip--
    {
       // --snip--
    } // The inner scope ends here, dropping the inner 'a' as it goes out of scope.
    
    println!("The value of a in the outer scope is: {a}");
}
```

we would encounter the following error during compilation.

```text
   Compiling scopes v0.1.0 (/var/home/z561/rust_projects/scopes)
error[E0425]: cannot find value `a` in this scope
  --> src/main.rs:11:54
   |
11 |     println!("The value of a in the outer scope is: {a}");
   |                                                      ^
   |
help: the binding `a` is available in a different scope in the same function
  --> src/main.rs:7:13
   |
 7 |         let a = x * 2; // 12 = 6 * 2
   |             ^

For more information about this error, try `rustc --explain E0425`.
error: could not compile `scopes` (bin "scopes") due to 1 previous error
```

Consequently, some questions should naturally arise:

- <span style={{ color: "#FF8C00" }}><b>Q-1</b>: Why shadow a variable by reusing the same name if it is actually a totally separate variable?</span>
- <span style={{ color: "#FF8C00" }}><b>Q-2</b>: Why should we even use bare scopes?</span>
- <span style={{ color: "#FF8C00" }}><b>Q-3</b>: How can we extract reusable results from a scope if every variable declared within it gets dropped once the scope ends?</span>

We will answer each question in the following sections. Before doing so, however, we need to cover a couple of fundamental concepts and distinguish the **generic block scopes** we just explored from the **scopes introduced by functions**.

While functions will be analyzed in depth later on, we can already look at a couple of basic examples.

## Scope Structure

You can picture nested scopes like the layers of an onion:

```text
... -> { 3rd { 2nd { 1st } 2nd } 3rd } -> ...
```

The innermost scope (marked as `1st`) is the first to disappear when its block ends. Then the `2nd` one disappears, followed by the `3rd`, and so on. **This means that the variables within each of the scopes shown will cease to exist in the exact same order.**

As we saw earlier, **a simple inner scope can see variables from its outer scopes (we had access to the variable `x` defined in the outer scope)**. This is partly because they are still 'alive'. However, **the reverse is not true: outer scopes cannot see inside inner scopes**.

If you think about it, this makes perfect sense. First, an outer scope cannot 'capture' values from an inner scope because **when the outer scope starts running, the inner variables haven't even been initialized yet**. Furthermore, **once the inner scope finishes executing, its variables are dropped and cease to exist**. **Allowing the outer scope to access them afterward would mean referencing data that is no longer there—a scenario that Rust's safety rules strictly forbid!**

Finally, note that our previous example showed the simplest form of an inner scope—one created explicitly with bare `{ ... }` brackets. In general, **a new scope is created almost every time you see curly brackets**. For instance, **defining a function or calling a method introduces a new scope**. However, there is a crucial difference: **unlike bare blocks, functions and methods do not automatically see the variables of the scope that called them**. **The parameters you explicitly pass to the function are the only outer variables allowed into that new scope.**

Let's dive into a simple function to see how variables are passed into its scope. Afterward, we will highlight the key differences between function scopes and the bare blocks we explored earlier.

Once that's done, we will finally answer the questions raised earlier, definitively clearing up the topic of scopes—a fundamental concept in Rust. This is exactly why I dedicated an entire section to it. Grasping how scopes work right from the start will massively accelerate and improve your understanding of Rust's more complex core mechanisms, such as its famous **ownership model**.

## Function Scopes

Let's start with an example of a simple function that takes a 32-bit unsigned integer as a parameter and returns nothing. The function merely prints the passed value `x` to the terminal.

```rust
fn print_value(x: u32) {
    println!("{x}");
}
```

When compiling and running the following code

```rust
fn main() {
    print_value(17);
}

fn print_value(x: u32) {
    println!("{x}");
}
```

the output will be as follows:

```
   Compiling scopes v0.1.0 (/var/home/z561/rust_projects/scopes)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.07s
     Running `target/debug/scopes`
17
```

What might not be immediately obvious when first looking at functions (methods are a specific type of function that follow the same rules regarding scopes, so we won't cover them separately) is that they introduce a nested scope, exactly like the bare block scopes discussed earlier.

The preceding code is related, but not strictly equivalent, to the following snippet:

```rust
fn main() {
    let x: u32 = 17;
    {
        println!("{x}");
    }
}
```

At this point, to demonstrate the concept of nested scopes in a less trivial way, let's define a new function, `inner_print`, that does the exact same thing as `print_value`, but performs the work from inside `print_value`.

```rust
fn main() {
    print_value(17);
}

fn print_value(x: u32) {
    // Nested function definition.
    fn inner_print(x: u32) {
        println!("{x}");
    }

    // Calling the nested function with the outer parameter.
    inner_print(x);
}
```

The preceding code is related, but not strictly equivalent, to the following one.

```rust
fn main() {
    let x: u32 = 17;
    {
        {
            println!("{x}");
        }
    }
}
```

At the same time, it is also true that the following code produces the same result as the preceding one:

```rust
fn main() {
    let x: u32 = 17;
    println!("{x}");
}
```

We are now ready to address <span style={{ color: "#FF8C00" }}><b>Q-1</b></span> and <span style={{ color: "#FF8C00" }}><b>Q-2</b></span>.

## Q&A Pt. 1

- <span style={{ color: "#FF8C00" }}><b>Q-2</b>: Why should we even use bare scopes?</span>
- <span style={{ color: "#0050FF" }}><b>A-2</b>: While it is entirely possible to use bare scopes in a completely pointless way, there are specific scenarios where they become highly beneficial. <b>Utilizing bare scopes is primarily useful when you want to discard specific modifications or temporary variables at the end of a set of operations.</b> This ensures that any variable defined within that scope—including those modified via shadowing—is automatically dropped once the scope ends, eliminating the need to track these variables later and ensuring their memory is released immediately.</span>
- <span style={{ color: "#FF8C00" }}><b>Q-1</b>: Why shadow a variable by reusing the same name if it is actually a totally separate variable?</span>
- <span style={{ color: "#0050FF" }}><b>A-1</b>: When defining an entirely new variable in a nested scope, <b>its only logical purpose is to serve as a temporary variable</b>. In the context of shadowing, the reason for reusing a name for this temporary variable is now clear: <b>the temporary variable is logically related to the original variable in the outer scope</b>. This is why I mostly referred to <b>temporary modifications</b> when discussing shadowing: the values of the shadowed variables are seamlessly restored as soon as the inner scope ends. Nevertheless, this behavior is merely an illusion. Under the hood, shadowed variables within inner scopes are entirely new variables, as demonstrated earlier.</span>

Building on the previous paragraph, we can now define and answer <span style={{ color: "#FF8C00" }}><b>Q-4</b></span> as follows.

- <span style={{ color: "#FF8C00" }}><b>Q-4</b>: If the preceding paragraph code snippets are related, why should we use bare scopes instead of functions, and viceversa? What are the actual differences between using functions and using bare scopes?</span>
- <span style={{ color: "#0050FF" }}><b>A-4</b>: There are primarily two differences between these two approaches.</span><br /><br />

  <span style={{ color: "#0050FF" }}>The first is that <b>bare scopes allow you to 'capture' all variables from the surrounding environment, unlike functions, which only allow you to 'capture' the parameters explicitly specified in their definition</b>. Take the following code as an example:</span>
  
  ```rust
  fn main() {
      let x: u32 = 17;
      let y: u32 = 1337;
      {
          {
              println!("{x}"); // Outputs '17', as expected.
              println!("{y}"); // Outputs '1337', as expected.
          }
      }
  }
  ```
  
  <span style={{ color: "#0050FF" }}>In this case, <b>had we used the functions shown previously, we would not have been able to print the variable `y` because it was not defined as a parameter in the `print_value` function!</b> Attempting to do so would result in a compilation error, as shown below.</span>
  
  ```rust
  fn main() {
      let x: u32 = 17;
      let y: u32 = 1337;
      print_value(x);
  }

  fn print_value(x: u32) {
      fn inner_print(x: u32) {
          println!("{x}");
      }

      inner_print(x);
      inner_print(y); /* error[E0425]: cannot find value `y` in this scope
                           --> src/main.rs:13:17
                            |
                         13 |     inner_print(y);
                            |                 ^
                            | */
  }
  ```
  
  <span style={{ color: "#0050FF" }}>The second difference between the two approaches is the more important one. Although <b>functions</b> are less 'open' in terms of variable capture, they <b>allow you to define much more precise and controllable workflows, enable code reuse, and lead to a significantly more readable and organized code structure</b>. For example, if we wanted to call `print_value` multiple times, we could do so simply by writing `print_value(value)`. <b>This behaviour would not be possible using bare scopes, as we would have to rewrite the code responsible for printing the values every single time.</b> While in this simple example using `println!` takes up roughly the same amount of space as calling `print_value`, functions typically contain complex logic and several lines of code. <b>To avoid duplicating that code over and over, we use named identifiers like `print_value`!</b> Ultimately, because of these factors, <b>the practical use case for bare scopes narrows down exclusively to the one explained earlier</b>. <b>In that specific scenario—namely, freeing memory from temporary variables—using a function instead would not be viable. This is because a function's parameters must be explicitly defined and have a fixed count.</b></span><br /><br />
  
  <span style={{ color: "#0050FF" }}>At this juncture, a further specification must be made. One might erroneously conclude that our previous assertion regarding fixed parameters is incorrect, given that <b>`println!` accepts an arbitrary number of them</b>. For instance, the following code will compile without any problems:</span>
  
  ```rust
  println!("{}, {}, {}, {}, ...", 1, 2, 3, 4); // Outputs '1, 2, 3, 4, ...'.
  ```
  
  <span style={{ color: "#0050FF" }}>The reality is that `println!` is not a function at all, but rather a <b>macro</b>. One of the major limitations of standard functions in Rust is that they require a fixed, explicit number of parameters. Macros bypass this limitation entirely, allowing them to accept an arbitrary number of arguments (though the fundamental purpose of macros is <b>metaprogramming</b>, and implementing them is significantly more complex than defining a standard function).</span>
  
## Returning Expression Results from Scopes

When the program exits a scope, all variables contained within it, get dropped and vanish. Fortunately, there is a way to preserve the results computed within a scope: **returning them implicitly or explicitly**.

### Expressions

As you will see later on, distinguishing between an **expression** and a variable is crucial in practically any programming language. Given the snippet

```rust
let x: u32 = 17;
```

`x` acts as a variable of type `u32`, while **`17` is an expression** made up of a single **numeric literal**. To clarify this further: if you write

```rust
fn main() {
    let x: u32 = 17; /* This statement is entirely equivalent to the following:
                        let x = 17_u32; */
    let y: u32 = 1337;

    let z = x + y;

    println!("{z}"); // Outputs '1354'.
}
```

`x`, `y`, and `z` serve as variables of type `u32`, whereas **`17`, `1337`, and `x + y` are evaluated as expressions**.

Explicitly inspecting a variable's type at runtime is somewhat complex in Rust. The code below is included only to let readers verify that `z` is indeed a `u32`. The underlying syntax and language mechanics will be thoroughly explored in later chapters.

```rust
fn print_type_of<T>(_: &T) {
    println!("{}", std::any::type_name::<T>());
}

fn main() {
    // --snip--
    print_type_of(&z); // Outputs 'u32'.
}
```

Beyond demonstrating how to print the type of any variable (which may prove useful later on), this example illustrates the utility of functions far better than `print_value`. To repeatedly print the types of different variables, we only need to write `print_type_of(&var)`. Conversely, if we were to use scopes, we would be forced to write out `{ println!("{}", std::any::type_name_of_val(&var)) }` each time, which makes little sense in this context, as it introduces a bare scope for absolutely no reason. If you actually try compiling code with that line, the Rust compiler will issue a warning indicating that the extra scope is entirely redundant. Nevertheless, the code still compiles and runs perfectly.

### Returning Results from Scopes

In Rust, you can explicitly return a result **from a function** using the `return` keyword. Alternatively, you can return a result **from any block scope** implicitly by simply omitting the semicolon `;` from the final expression inside that scope.

This means there is no explicit keyword to return values from a bare block scope, although there is an advanced technique that allows you to return values semi-explicitly from bare blocks. We will not analyze it in this section, but we will explore it shortly.

The following code provides a basic example of evaluating expressions and returning their results from bare blocks for assignments.

```rust
fn main() {
    let x: u32 = { 17 };
    let y: u32 = { 1337 };

    let z = { x + y };

    println!("{z}"); // Outputs '1354'.
}
```

By contrast, the next snippet demonstrates how to explicitly return a value from the `ret_explicitly` function, and how to implicitly return one from `ret_implicitly`.

```rust
fn ret_explicitly(x: u32) -> u32 {
    return x;
}

fn ret_implicitly(x: u32) -> u32 {
    x
}

fn main() {
    let x: u32 = ret_explicitly(17);
    let y: u32 = ret_implicitly(1337);

    let z = x + y;

    println!("{z}"); // Outputs '1354'.
}
```

## Bridging the Gap: Scopes and References

So far, we have utilized scopes to output basic values to the console or to return data passed into a function. However, what we have not yet addressed—which lies at the very core of Rust programming—is that **this approach breaks down if we do not utilize references (`&`) <span style={{ color: "#FFFF00" }}>with types that do not implement the `Copy` trait</span>, and if we intend to pass a value into a function while keeping that variable accessible in the outer scope <span style={{ color: "#FFFF00" }}>without an explicit reassignment via `let`</span>**.

Unfortunately (or perhaps fortunately), every scalar type in Rust implements the `Copy` trait, a concept we will explore in the next chapter. This means that **whenever a scalar variable is passed to a function or reassigned, it is effectively duplicated, generating an independent, bit-for-bit copy of its raw data in memory**. However, for the sake of memory efficiency and performance, this **automatic duplication does not apply to types that manage dynamic heap memory** (like `String`).

Therefore, to accurately demonstrate the ownership behavior we just discussed, we must use a type that does not implement `Copy`. The `String` type is the most prominent and commonly used example of this. You do not need to understand every detail of the following code just yet. The crucial takeaway is that our previous approach fails at compile time when applied to these non-`Copy` types, illustrating the exact ownership problem we outlined.

The snippet below illustrates this exact problem and displays the resulting compiler error.

```rust
fn drops_s_x_and_s_y(s_x: String, s_y: String) -> String {
    format!("{s_x} + {s_y}")
}

fn doesnt_drop_s_x_and_s_y(s_x: &String, s_y: &String, s_z: &mut String) {
    *s_z = format!("{s_x} + {s_y}");
}

fn main() {
    let s_x: String = String::from("17");
    let s_y: String = String::from("1337");
    // let mut s_z: String = String::new();

    let mut s_z: String = drops_s_x_and_s_y(s_x, s_y);

    // doesnt_drop_s_x_and_s_y(&s_x, &s_y, &mut s_z);

    // println!("{s_x}");
    // println!("{s_y}");
    println!("{s_z}"); // Outputs '17 + 1337'.
}
```

As mentioned earlier, you do not need to understand every single line of this code just yet. The key takeaway is that this snippet defines three `String` variables: `s_x`, `s_y`, and `s_z`. We pass only `s_x` and `s_y` into the first function, `drops_s_x_and_s_y`, which computes the result—the string `17 + 1337`—and returns it to be assigned to `s_z`. So far, everything works perfectly! If you execute the preceding code using `cargo run`, the program will compile and run successfully. While the compiler will issue a few warnings, the execution completes normally, outputting the result `17 + 1337` to the console.

However, if we now uncomment the `println!("{s_x}")` (or `println!("{s_y}")`) statement, we observe the following output:

```text
error[E0382]: borrow of moved value: `s_x`
  --> src/main.rs:18:16
   |
10 |     let s_x: String = String::from("17");
   |         --- move occurs because `s_x` has type `String`, which does not implement the `Copy` trait
...
14 |     let mut s_z: String = drops_s_x_and_s_y(s_x, s_y);
   |                                             --- value moved here
...
18 |     println!("{s_x}");
   |                ^^^ value borrowed here after move
   |
note: consider changing this parameter type in function `drops_s_x_and_s_y` to borrow instead if owning the value isn't necessary
  --> src/main.rs:1:27
   |
 1 | fn drops_s_x_and_s_y(s_x: String, s_y: String) -> String {
   |    -----------------      ^^^^^^ this parameter takes ownership of the value
   |    |
   |    in this function
help: consider cloning the value if the performance cost is acceptable
   |
14 |     let mut s_z: String = drops_s_x_and_s_y(s_x.clone(), s_y);
   |                                                ++++++++

For more information about this error, try `rustc --explain E0382`.
warning: `scopes` (bin "scopes") generated 1 warning
error: could not compile `scopes` (bin "scopes") due to 1 previous error; 1 warning emitted
```

In this instance, the compiler flags an error indicating that the value `s_x` has been '**moved**', and that we subsequently attempted to '**borrow**' `s_x` after this '**move**' took place. For now, we can gloss over the compiler's specific mention of **ownership**, although, as hinted earlier, it is the underlying theme of this entire section. The compiler also explicitly notes that `s_x` is of type `String`, which does not implement the `Copy` trait, just as we established previously.

To summarize, **the issue arises because we are passing `s_x` into the `drops_s_x_and_s_y` function**. By doing this, **the value is moved (transferring ownership) and is effectively destroyed**—or '**dropped**'—**when the function's scope ends**. **As a result, it is completely invalid to reference or use it in any way back in the `main` function's scope.**

Conversely, if we utilize `doesnt_drop_s_x_and_s_y` instead (by uncommenting the line `doesnt_drop_s_x_and_s_y(&s_x, &s_y, &mut s_z)`), you will see that we can also safely uncomment both `println!` macros. If you run the code below with `cargo run`, it will compile perfectly without throwing any errors.

```rust
fn drops_s_x_and_s_y(s_x: String, s_y: String) -> String {
    format!("{s_x} + {s_y}")
}

fn doesnt_drop_s_x_and_s_y(s_x: &String, s_y: &String, s_z: &mut String) {
    *s_z = format!("{s_x} + {s_y}");
}

fn main() {
    let s_x: String = String::from("17");
    let s_y: String = String::from("1337");
    let mut s_z: String = String::new();

    // let mut s_z: String = drops_s_x_and_s_y(s_x, s_y);

    doesnt_drop_s_x_and_s_y(&s_x, &s_y, &mut s_z);

    println!("{s_x}"); // Outputs '17'.
    println!("{s_y}"); // Outputs '1337'.
    println!("{s_z}"); // Outputs '17 + 1337'.
}
```
