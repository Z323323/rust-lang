Generics allow you to write a single piece of code that works for many different concrete types, eliminating duplication without sacrificing type safety. If you've ever written two functions that do the same thing but for `i32` and `f64` respectively, generics are the answer. Rust's generic system is powerful, zero-cost (see *monomorphization* below), and used pervasively throughout the standard library — `Vec<T>`, `Option<T>`, `Result<T, E>`, and `HashMap<K, V>` are all generic types.

## Removing Duplication with Generics

Consider these two functions that find the largest number in a slice:

```rust
fn largest_i32(list: &[i32]) -> &i32 {
    let mut largest = &list[0];
    for item in list {
        if item > largest { largest = item; }
    }
    largest
}

fn largest_f64(list: &[f64]) -> &f64 {
    let mut largest = &list[0];
    for item in list {
        if item > largest { largest = item; }
    }
    largest
}
```

They're identical except for the type. We can merge them into one generic function:

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {
    let mut largest = &list[0];
    for item in list {
        if item > largest { largest = item; }
    }
    largest
}

fn main() {
    let numbers = vec![34, 50, 25, 100, 65];
    println!("Largest number: {}", largest(&numbers));

    let chars = vec!['y', 'm', 'a', 'q'];
    println!("Largest char: {}", largest(&chars));
}
```

`T: PartialOrd` is a **trait bound** — it tells the compiler that `T` must support the `>` operator. Without it, the compiler would refuse to compile the comparison, because not every type supports ordering.

## Generic Structs

You can parameterize a struct over one or more type parameters:

```rust
struct Point<T> {
    x: T,
    y: T,
}

fn main() {
    let integer_point = Point { x: 5, y: 10 };
    let float_point   = Point { x: 1.0, y: 4.0 };

    println!("integer_point: ({}, {})", integer_point.x, integer_point.y);
    println!("float_point:   ({}, {})", float_point.x,   float_point.y);

    // This would NOT compile — x and y must be the same type T
    // let mixed = Point { x: 5, y: 4.0 };
}
```

To allow `x` and `y` to be *different* types, use two type parameters:

```rust
struct Point<T, U> {
    x: T,
    y: U,
}

fn main() {
    let mixed = Point { x: 5, y: 4.0 }; // T = i32, U = f64
    println!("mixed: ({}, {})", mixed.x, mixed.y);
}
```

## Generic Enums

You've already been using generic enums without thinking about it. `Option<T>` and `Result<T, E>` are both defined this way in the standard library:

```rust
// How Option is actually defined:
enum Option<T> {
    Some(T),
    None,
}

// How Result is actually defined:
enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

You can define your own:

```rust
#[derive(Debug)]
enum Tree<T> {
    Leaf(T),
    Node(Box<Tree<T>>, Box<Tree<T>>),
}

fn main() {
    let t = Tree::Node(
        Box::new(Tree::Leaf(1)),
        Box::new(Tree::Leaf(2)),
    );
    println!("{:?}", t);
}
```

## Generic Methods with `impl<T>`

To define methods on a generic struct, you must declare the type parameter after `impl`:

```rust
struct Point<T> {
    x: T,
    y: T,
}

impl<T> Point<T> {
    // Works for any T
    fn x(&self) -> &T {
        &self.x
    }
}

impl Point<f64> {
    // Only available when T is f64
    fn distance_from_origin(&self) -> f64 {
        (self.x * self.x + self.y * self.y).sqrt()
    }
}

fn main() {
    let p_int   = Point { x: 5, y: 10 };
    let p_float = Point { x: 3.0_f64, y: 4.0_f64 };

    println!("p.x = {}", p_int.x());
    println!("distance = {}", p_float.distance_from_origin()); // 5.0
    // p_int.distance_from_origin() — COMPILE ERROR, not defined for i32
}
```

The `impl<T>` block applies to all `Point<T>`, while `impl Point<f64>` adds methods only for the concrete `f64` specialization.

## Monomorphization: Zero-Cost Generics

Rust uses **monomorphization** to implement generics. At compile time, Rust inspects every concrete type that a generic function or struct is instantiated with and generates dedicated machine code for each one. There is no runtime overhead — it's as if you had written the type-specific versions by hand.

```rust
fn double<T: std::ops::Add<Output = T> + Copy>(x: T) -> T {
    x + x
}

fn main() {
    // The compiler generates separate `double_i32` and `double_f64` functions
    println!("{}", double(5_i32));
    println!("{}", double(3.14_f64));
}
```

The compiled binary contains two distinct machine-code functions. This is fundamentally different from how generics work in languages like Java (type erasure) or Go (interface boxing), both of which have runtime overhead. In Rust, **you pay nothing at runtime for using generics**.

## Summary

| Location | Syntax |
|---|---|
| Generic function | `fn foo<T>(x: T) -> T { … }` |
| Generic struct | `struct Foo<T> { … }` |
| Generic enum | `enum Foo<T> { … }` |
| `impl` block for all T | `impl<T> Foo<T> { … }` |
| `impl` block for one type | `impl Foo<i32> { … }` |
| With trait bounds | `fn foo<T: Display + Clone>(x: T)` |
