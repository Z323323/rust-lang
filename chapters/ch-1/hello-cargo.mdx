>Cargo is Rust’s build system and package manager. Cargo handles a lot of tasks for you, such as building your code, **downloading** the libraries your code depends on, and building those libraries.

## Creating a Project with Cargo

<CodeGroup>
```bash Command 1.3.1
cd ~/rust_projects &&
# Output 1.3.1.1
cargo new hello_cargo &&
cd hello_cargo &&
# Output 1.3.1.2
ls -a &&
cd src &&
# Output 1.3.1.3
ls &&
# Output 1.3.1.4
cat main.rs
```
```text Output 1.3.1.1
    Creating binary (application) `hello_cargo` package
note: see more `Cargo.toml` keys and their definitions at https://doc.rust-lang.org/cargo/reference/manifest.html
```
```text Output 1.3.1.2
.  ..  Cargo.toml  .git  .gitignore  src
```
```text Output 1.3.1.3
main.rs
```
```rust Output 1.3.1.4
fn main() {
    println!("Hello, world!");
}
```
</CodeGroup>

<Steps>
  <Step title={<span className="code-comment"># Output 1.3.1.[1, 2]</span>}>
  >`cargo new` creates a new directory and project called _hello_cargo_. We’ve named our project _hello_cargo_, and Cargo creates its files in a directory of the same name.
  
  >You’ll see that Cargo has generated two files and one directory for us: a _Cargo.toml_ file and a _src_ directory with a _main.rs_ file inside.
  
  >It has also initialized a new Git repository along with a _.gitignore_ file. Git files won’t be generated if you run `cargo new` within an existing Git repository; you can override this behavior by using `cargo new --vcs=git`. You can change `cargo new` to use a different version control system or no version control system (`none`) by using the `--vcs` flag.
  </Step>
  <Step title={<span className="code-comment"># Output 1.3.1.[3, 4]</span>}>
  >Cargo has generated a “Hello, world!” program for you. So far, the differences between our project (_hello_world_) and the project Cargo generated (_hello_cargo_) are that Cargo placed the code in the _src_ directory, and we have a _Cargo.toml_ configuration file in the top directory.
  
  >**Cargo expects your source files to live inside the _src_ directory. The top-level project directory is just for README files, license information, configuration files, and anything else not related to your code. Using Cargo helps you organize your projects.** There’s a place for everything, and everything is in its place.
  
  >If you started a project that doesn’t use Cargo, as we did with the “Hello, world!” project, you can convert it to a project that does use Cargo. Move the project code into the _src_ directory and create an appropriate _Cargo.toml_ file. One easy way to get that _Cargo.toml_ file is to run `cargo init`, which will create it for you automatically.
  </Step>
</Steps>

_Cargo.toml_ is the configuration file used to specify your project's version, alongside various other details and package dependencies (and their versions).

<CodeGroup>
```bash Command 1.3.2
cd .. &&
# Output 1.3.2
cat Cargo.toml
```
```toml Output 1.3.2
[package]
name = "hello_cargo"
version = "0.1.0"
edition = "2024"

[dependencies]
```
</CodeGroup>

>The first line, `[package]`, is a section heading that indicates that the following statements are configuring a package. As we add more information to this file, we’ll add other sections.

>The next three lines set the configuration information Cargo needs to compile your program: the name, the version, and the **edition** of Rust to use.

>The last line, `[dependencies]`, is the start of a section for you to list any of your project’s dependencies. In Rust, packages of code are referred to as _crates_.

## Building and Running a Cargo Project

<CodeGroup>
```bash Command 1.3.3
# Output 1.3.3.1
cargo build &&
# Output 1.3.3.2
./target/debug/hello_cargo
```
```text Output 1.3.3.1
   Compiling hello_cargo v0.1.0 (/var/home/z561/rust_projects/hello_cargo)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.34s
```
```text Output 1.3.3.2
Hello, world!
```
</CodeGroup>

<Steps>
  <Step title={<span className="code-comment"># Output 1.3.3.1</span>}>
>This command creates an executable file in _target/debug/hello_cargo_ rather than in your current directory. Because the default build is a debug build, Cargo puts the binary in a directory named _debug_.
  
>Running `cargo build` for the first time also causes Cargo to create a new file at the top level: **_Cargo.lock_**. **This file keeps track of the exact versions of dependencies in your project.** This project doesn’t have dependencies, so the file is a bit sparse. You won’t ever need to change this file manually; Cargo manages its contents for you.
  </Step>
  <Step title={<span className="code-comment"># Output 1.3.3.2</span>}>
Executing `./target/debug/hello_cargo` runs the program built by `cargo build`.
  </Step>
</Steps>

>**We can also use `cargo run` to compile the code and then run the resultant executable all in one command.**

<CodeGroup>
```bash Command 1.3.4
cargo run
```
```text Output 1.3.4
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.01s
     Running `target/debug/hello_cargo`
Hello, world!
```
</CodeGroup>

>This time we didn’t see output indicating that Cargo was compiling _hello_cargo_. Cargo figured out that the files hadn’t changed, so it didn’t rebuild but just ran the binary.

>Cargo also provides a command called `cargo check`. This command quickly checks your code to make sure it compiles but doesn’t produce an executable.

<CodeGroup>
```bash Command 1.3.5
cargo check
```
```text Output 1.3.5
    Checking hello_cargo v0.1.0 (/var/home/z561/rust_projects/hello_cargo)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.02s
```
</CodeGroup>

>Why would you not want an executable? Often, `cargo check` is much faster than `cargo build` because it skips the step of producing an executable.

<Note>
**From this point forward, to minimize unnecessary repetition and keep the sections concise, most code listings (labeled as Code X.Y.Z) will be directly accompanied by their corresponding output**, just like the terminal commands.

To make things perfectly clear, we will use a specific naming convention for these outputs:

- **C-Output X.Y.Z (Compilation): This label indicates that the output is a direct result of the compilation process (`cargo check`, `cargo build`, `cargo run`). In almost all cases, this means it will display a compile-time error emitted by `rustc`!**

- **E-Output X.Y.Z (Execution): This label indicates that the output is generated by the actual execution of the program (`cargo run`). The 'E' stands for Execution (NOT Error). Typically, when you see this, you can assume the code compiled and ran without errors—though keep in mind there will be occasional exceptions!**

In the upcoming sections and chapters, we will gradually phase out the approach of showing explicit terminal commands and their raw outputs, transitioning entirely to the streamlined convention we just described.
</Note>

## Building for Release

>When your project is finally ready for release, you can use `cargo build --release` to compile it with optimizations. This command will create an executable in _target/release_ instead of _target/debug_. The optimizations make your Rust code run faster, but turning them on lengthens the time it takes for your program to compile. This is why there are two different **profiles**: one for development, when you want to rebuild quickly and often, and another for building the final program you’ll give to a user that won’t be rebuilt repeatedly and that will run as fast as possible. If you’re benchmarking your code’s running time, be sure to run `cargo build --release` and benchmark with the executable in _target/release_.
