>Ownership is a set of rules that govern how a Rust program manages memory. If any of the rules are violated, the program won’t compile. None of the features of ownership will slow down your program while it’s running.

>In this chapter, you’ll learn ownership by working through some examples that focus on a very common data structure: strings.<br />

**The Stack and the Heap**

>In a systems programming language like Rust, whether a value is on the stack or the heap affects how the language behaves and why you have to make certain decisions.

><p>Both the stack and the heap are parts of memory available to your code to use at runtime, but they are structured in different ways. The stack stores values in the order it gets them and removes the values in the opposite order. This is referred to as <u>l</u>ast <u>i</u>n, <u>f</u>irst <u>o</u>ut (LIFO). Adding data is called <u>pushing onto the stack</u>, and removing data is called <u>popping off the stack</u>. <u>All data stored on the stack must have a known, fixed size.</u></p>
>   <div style={{ display: 'flex', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↘</span>
>     <div>
>      <p>All of the scalar types which we explored in section 3.3 have a known, fixed size at compile time. Because of this, <u>they are stored directly on the stack</u> and inherently implement the `Copy` trait! Regarding the types discussed in section 3.3: 'Tuples implements `Copy` if they only contain types that also implement `Copy`. For example, `(i32, i32)` implements `Copy`, but `(i32, String)` does not'. Arrays can also contain elements that do not implement the `Copy` trait. In such cases, arrays exhibit the exact same behavior as tuples.</p>
>     </div>
>   </div>

><p><u>Data with an unknown size at compile time or a size that might change must be stored on the heap instead. When you put data on the heap, you request a certain amount of space. The memory allocator finds an empty spot in the heap that is big enough, marks it as being in use, and returns a pointer, which is the address of that location.</u> This process is called <u>allocating on the heap</u> (pushing values onto the stack is not considered allocating). <u>Because the pointer to the heap is a known, fixed size, you can store the pointer on the stack, but when you want the actual data, you must follow the pointer.</u></p>
>   <div style={{ display: 'flex', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↘</span>
>     <div>
>      <p>Types that manage dynamically sized data (such as `String`) must allocate their underlying memory on the heap. Consequently, they cannot implement the `Copy` trait.</p>
>     </div>
>   </div>

><p><u>Pushing to the stack is faster than allocating on the heap because the allocator never has to search for a place to store new data; that location is always at the top of the stack. Comparatively, allocating space on the heap requires more work because the allocator must first find a big enough space to hold the data and then perform bookkeeping to prepare for the next allocation.</u></p>
>   <div style={{ display: 'flex', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↘</span>
>     <div>
>      <p>Remember, main memory access (I/O) operations are slower than the processor's calculation speed—often taking up to 100 to 300 times longer than a single CPU cycle. If the data the CPU needs to process is grouped in contiguous memory chunks (as it is on the stack), the hardware's prefetcher can efficiently load this data into the extremely fast cache memories (L1, L2, and L3). This guarantees a massive performance boost, making data retrieval roughly 10 to 100 times faster by completely bypassing the RAM. To give you an idea of how critical this optimization is, consider that L1 caches are physically etched directly onto the silicon die, often positioned just fractions of a millimeter away from the execution cores. To achieve these microscopic latency times, the caches themselves must be physically tiny; an L1 cache block typically occupies a mere 0.1 to 0.2 square millimeters (mm²) of silicon, while a larger shared L3 cache spans roughly 15 to 20 mm². In stark contrast, RAM is located centimeters away on the motherboard. Because data transfer is bound by the physical speed of electrical signals, the processor cannot leverage its blazing speed if it is forced to jump around main memory in a semi-random fashion to gather scattered data—which is exactly what happens with data allocated on the heap.</p>
>     <div style={{ display: 'flex', color: '#009B77', marginTop: '-1.2rem', marginLeft: '-0.8rem' }}>
>       <span>↗</span>
>     </div>
>     </div>
>   </div>
><p><u>Accessing data in the heap is generally slower than accessing data on the stack because you have to follow a pointer to get there. Contemporary processors are faster if they jump around less in memory. A processor can usually do its job better if it works on data that’s close to other data (as it is on the stack) rather than farther away (as it can be on the heap).</u></p>

><p><u>When your code calls a function, the values passed into the function (including, potentially, pointers to data on the heap) and the function’s local variables get pushed onto the stack. When the function is over, those values get popped off the stack.</u></p>
>   <div style={{ display: 'flex', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↘</span>
>     <div>
>      <p>The exact same thing, actually, applies to all the scopes we looked at in section 3.2 as well. In fact, whenever an inner scope ends, all the variables defined within it are dropped as they are popped off the stack. It's all starting to make more sense now, isn't it?</p>
>     </div>
>   </div>

>Keeping track of what parts of code are using what data on the heap, minimizing the amount of duplicate data on the heap, and cleaning up unused data on the heap so that you don’t run out of space are all problems that ownership addresses. Once you understand ownership, you won’t need to think about the stack and the heap very often. But knowing that the main purpose of ownership is to manage heap data can help explain why it works the way it does.
</Callout>

## Ownership Rules

>First, let’s take a look at the ownership rules. Keep these rules in mind as we work through the examples that illustrate them:

>- Each value in Rust has an **owner**.<br />
>   <div style={{ display: 'flex', gap: '8px', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↳</span>
>     <div>
>       While this fact is true, it can cause confusion if we do not immediately distinguish between variables of types that implement the `Copy` trait and those that do not. Generally speaking, the fact that every value has an owner is fairly obvious: from a programmer's perspective, practically every value must be bound to a variable (its owner), otherwise, that specific value would be unusable. However, starting to view the concept of ownership as counter-intuitive for every variable is, in my opinion, incorrect and leads to confusion due to how the `Copy` trait works. As noted in section 3.2, at the end of the day, the only mechanism you truly need to worry about is the `=` assignment for variables whose types <u>do not</u> implement `Copy`. This is because the variable <u>on the RIGHT side of the equals sign</u> (specifically, the memory address it points to) is <u>moved</u> if we do not apply the `&` symbol to explicitly <u>copy</u> its address. When this happens, it permanently <u>loses ownership of both its stack data (the memory address pointing to the heap) and the underlying heap memory itself</u>. In these specific cases, <u>the LEFT variable gains ownership of the heap memory address and the data it points to</u>. If the RIGHT variable were to be destroyed at the end of the scope in which it is defined, then the ownership would be permanently lost and the data destroyed (both the heap address on the stack, and the data on the heap). <u>In any case, the EXISTING data will always have an owner.</u>
>     </div>
>   </div>
>- There can only be <u>one</u> owner at a time.<br />
>   <div style={{ display: 'flex', gap: '8px', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↳</span>
>     <div>
>       This concept aligns seamlessly with our previous observations and does not conflict with the underlying mechanics of types implementing the `Copy` trait. When a value belonging to one variable is copied to another, there must necessarily exist <u>TWO distinct copies of that value in memory, bound to TWO distinct variables (the owners)</u>. This principle remains valid even when the values being copied are identical memory addresses referencing the same data on the heap. It applies equally to the stack: <u>copying memory addresses that point to stack data is entirely possible using references</u>. <u>Ultimately, the `Copy` trait simply performs a direct copy of the value's bits—whether those bits represent the actual data itself or just an address pointing to it.</u> Furthermore, the restriction that a value may have only a single owner at any given time does not preclude the possibility of data losing its ownership entirely and being deallocated from memory, as demonstrated in section 3.2.
>     </div>
>   </div>
>- When the owner goes out of scope, the value will be dropped.<br />
>   <div style={{ display: 'flex', gap: '8px', marginTop: '0.5rem', color: '#009B77' }}>
>     <span>↳</span>
>     <div>
>       This absolute rule directly reflects our observations from section 3.2 and needs no further breakdown.
>     </div>
>   </div>

<ol className="list-comments">
 <li>
 The statement `let mut s1 = String::with_capacity(10)` declares `s1` as a mutable `String` with an explicitly defined `capacity` of `10`. We do this to ensure absolute certainty about the memory fields we are going to access. If the `capacity` and the `len` (length) were identical, distinguishing between the two values in raw memory would require overly complex techniques.
 </li>
 <li>
 `s1.push_str("hello")` appends the string `hello` to `s1`. At this point, `s1`'s internal data perfectly matches what is shown in Figure 4.4.1, with the sole exception of the `capacity`. Note that if we hadn't declared `s1` as mutable, modifying it after creation would have been impossible!
 </li>
 <li>
 The complexity of the statement `let ptr = &s1 as *const String as *const usize` lies entirely on the right side of the `=` sign, as the left side simply declares an immutable variable named `ptr`.

The expression `&s1 as *const String as *const usize` starts by retrieving `s1`'s memory address on the STACK. It is important to remember that <u>`s1` is not merely the string data residing on the HEAP</u>, as shown in Figure 4.4.1. In fact, the address obtained by `&s1` points to the very first piece of data in memory belonging to the LEFT-hand structure depicted in Figure 4.4.1. In reality, under the Rust compiler's actual memory management, the data layout is not ordered exactly as shown in Figure 4.4.1. In fact, as Code 4.4.1 demonstrates, the first value residing in memory at `s1`'s address is the `capacity`'s one, not the pointer to the string data on the HEAP! Uncovering this specific structural quirk actually took quite a bit of effort.

Returning to the line of code in question: once we obtain the address of `s1`'s STACK data structure, `as *const String` acts as an explicit **type cast**. It allows us to treat `&s1` as a raw pointer to a `String` data structure (which is exactly what it represents). <u>This initial step is required before we can perform the second</u> cast, `as *const usize`. This final cast is necessary so we can operate on it as a <u>generic raw pointer</u>.
 </li>
 <li>
 The `unsafe { ... }` block encompassing almost all of Code 4.4.1 is strictly necessary to operate directly at the raw memory level, preventing the Rust compiler from stepping in and stopping us <u>for safety reasons</u>! Generally, <u>using `unsafe` is discouraged unless absolutely required</u>. However, this is one of those exceptions: to explore what truly happens under the hood and manipulate memory addresses, we have to tell the compiler that we know what we're doing, by reassuring it with an `unsafe` block. Don't worry, though! Later on, we will see exactly how to analyze low-level memory using a completely safe approach.

There is one more caveat to add. In idiomatic Rust, `unsafe` should be scoped as narrowly as possible; writing massive `unsafe` blocks like the one in Code 4.4.1 is generally considered bad practice (we will explain exactly why in Ch. 20). In this specific instance, it is structured this way simply because literally almost every single line of code inside that block performs an unsafe operation 😂.
 </li>
 <li>
 The line `let capacity = *ptr` defines the `capacity` variable by **dereferencing** the generic raw pointer `ptr` that we defined in `#3`!
 
 <u>The dereference operator (`*`) allows you to access the value stored in memory at a specific address.</u> In this case, the pointer `ptr` holds the memory address that—as noted earlier—points to the string's `capacity` field. This means that <u>the expression `*ptr` evaluates to `10`</u>, and this value is bound to `capacity` via `let`.

 <u>Dereferencing is the exact inverse of referencing (`&`).</u> In fact, writing `*&ptr` is functionally identical to simply writing `ptr`; a rule that holds true for any data type.

 Dereferencing a <u>raw pointer</u> is an unsafe operation! Attempting to write `*ptr` outside of an `unsafe` block will trigger a compiler error.

 That being said, dereferencing a standard reference (`&`)—rather than a raw pointer—is a perfectly safe operation in Rust! For example, Code 4.4.2 compiles and runs smoothly.

 ```rust Code 4.4.2
 fn main() {
     let x = 17;
     let y = &x; // 'y' is a standard, safe reference to 'x'.

     // We can safely dereference 'y' using '*' without an 'unsafe' block!
     let z = *y; 

     println!("{z}"); // Prints: '17'.
 }
 ```
 
 At this point, a natural question arises:
 
 <span style={{ color: "#964B00" }}><b>Q</b>: Why is dereferencing a standard reference considered safe, while dereferencing a raw pointer is considered dangerous?</span><br />
 <span style={{ color: "#008080" }}><b>A</b>: The answer primarily lies in the different <u>operations</u> permitted on a raw pointer versus a standard reference. As we are going to see in `#6`, and as clearly demonstrated in Code 4.4.1, <u>you can perform pointer arithmetic on a raw pointer</u>. <u>When you subsequently dereference that shifted pointer, you can access memory areas that might be completely unallocated or belong to entirely different variables. If not handled with extreme care, this capability can trigger severe vulnerabilities and undefined behavior—things Rust traditionally works very hard to prevent.</u></span><br />

 <span style={{ color: "#008080" }}>Despite these risks, Rust provides the `unsafe` keyword to guarantee maximum flexibility for low-level scenarios that require it, trusting the programmer only when they are absolutely certain of what they are doing.</span><br />

 <span style={{ color: "#008080" }}>The story is completely different for standard references created with the `&` operator. First and foremost, <u>you cannot perform pointer arithmetic on a standard reference</u>! <u>This single restriction prevents you from accessing memory outside the strict control of the compiler.</u> Because you cannot mathematically alter the address, it logically follows that:</span><br /><br />

 <span style={{ color: "#008080" }}>1. A reference can only point to existing, valid variables (meaning data that has not been dropped at the end of an inner scope).</span><br />
<span style={{ color: "#008080" }}>2. <u>You simply cannot use one variable's reference to magically peek into memory belonging to another!</u></span><br /><br />

 <span style={{ color: "#008080" }}>The ownership mechanisms we observed in section 3.2 naturally <u>prevent access to freed or unowned memory</u>, eliminating vulnerabilities and unexpected behavior. This is exactly why Rust correctly identifies standard references (`&`) as risk-free and inherently safe.</span><br />

 <span style={{ color: "#008080" }}>However, as we will explore later, <u>allowing multiple variables to hold references pointing to the exact same memory location can create complex scenarios that require detailed analysis.</u></span>
 </li>
 <li>
 The statement `let ptr_to_heap = *(ptr.add(1))` works much like the previous line. The key difference is that before accessing the value via `*`, we use `.add(1)` to shift the memory address held in `ptr`. This allows us to access the specific STACK address that points directly to the HEAP data containing our `hello` string. This process serves as a perfect example of the unsafe operation (pointer arithmetic) we discussed earlier. Thinking back to step `#4`, a quick glance at Code 4.4.1 reveals that it is packed with pointer arithmetic. This is exactly why the entire snippet had to be wrapped in a single, massive `unsafe` block.

 At this point, we must highlight a few fundamental details:

 - It is important to remember that while <u>a single memory address technically points to one byte (8 bits) of data</u>, <u>the `.add(1)` operation uses pointer arithmetic</u>. <u>Since we are operating on a `*const usize` pointer, adding 1 moves the pointer forward by the size of one complete `usize` (usually 8 bytes)</u>, properly aligning us with the next chunk of data in the `s1` structure on the STACK. If you look closely at E-Output 4.4.1, you will notice that `0x7ffe70bccbb8` (the address of the `ptr_to_heap` field) is located exactly 8 bytes after `0x7ffe70bccbb0` (the address of the `capacity` field).

   ```math
   \mathtt{7ffe70bccbb0 + 8 = 7ffe70bccbb8}
   ```

   These 8 bytes—or 64 bits—perfectly match the size of a `usize`, which corresponds to a single CPU register on a 64-bit architecture. Seeing these specific values printed shortly after in E-Output 4.4.1 practically proves the pointer arithmetic concepts we just discussed.

 - The previous point also explains the quirks we observed in step `#3`: when applying the `.add(x)` method to a `*const String` pointer, the address actually shifts by `24 * x` bytes. This is because <u>the compiler automatically multiplies the addition by the size of the referenced type</u>. In this case, <u>a `String` structure (Figure 4.4.1) occupies 24 bytes (comprising three 8-byte fields: a heap pointer, length, and capacity)</u>. This is exactly why we performed an additional cast to `*const usize` instead of keeping `*const String` as the raw pointer type in `#3`!

 - You can visually verify each address is indeed 8-byte size by looking at a memory address format, such as `0x7ffe70bccbb0`. <u>While it looks like 12 characters (excluding the `0x` hexadecimal prefix), the true underlying 64-bit value is `00007ffe70bccbb0`. When you count those digits, you get exactly 16 hexadecimal characters. Because each hex digit is 4 bits, 16 digits yield 64 bits, which is exactly 8 bytes per field.</u>

   ```math
   \mathtt{\underbrace{\underbrace{0}_{4}\underbrace{0}_{4}\underbrace{0}_{4}\underbrace{0}_{4}\underbrace{7}_{4}\underbrace{f}_{4}\underbrace{f}_{4}\underbrace{e}_{4}\underbrace{7}_{4}\underbrace{0}_{4}\underbrace{b}_{4}\underbrace{c}_{4}\underbrace{c}_{4}\underbrace{b}_{4}\underbrace{b}_{4}\underbrace{0}_{4}}_{64 \text{ bits} = 8 \text{ bytes}}}
    ```
 </li>
 <li>
 The line `println!("[Address of capacity (on the STACK)]: {:p}", &capacity)` simply prints the stack memory address of the `capacity` variable. To achieve this, we must use the `:p` **format specifier**, which instructs the `println!` macro to format the output as a memory address (a pointer). This step is crucial for two main reasons:
 
 - First, it allows us to directly observe the address, which—alongside the addresses of `ptr_to_heap` and `len`—made our pointer arithmetic analysis in `#6` possible.

 - The second reason is tied to the numerical structure of stack addresses themselves. If you compare them, you will notice that <u>stack addresses are numerically much higher than heap addresses</u>. This is no coincidence; it is a direct consequence of how memory is structured at a low level by the operating system.
 
   Figure 4.4.2 illustrates how <u>stack and heap memory grow in opposite directions</u>. <u>The stack begins near the highest memory addresses and grows downward (towards lower addresses). In contrast, the heap starts near lower memory addresses and grows upward (towards higher addresses, except when dynamically filling memory gaps).</u>

   For this reason, our earlier statement in `#3`—'the first value residing in memory at `s1`’s address is the `capacity`’s one'—might seem debatable. Because `capacity` sits at the lowest memory address compared to `ptr_to_heap` and `len`, it was physically the last field pushed onto the stack.

   While this is true regarding allocation order, <u>memory blocks are traditionally read from the lowest address upward</u>. Therefore, <u>the last item pushed onto the downward-growing stack logically becomes the 'first' item encountered when reading that memory block</u>. It is no coincidence that the base pointer `ptr` points exactly to `capacity`!
