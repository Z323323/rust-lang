Functions in Rust are first-class values. Beyond accepting closures via generic bounds or `dyn Fn`, Rust has a distinct `fn` pointer type that lets you store and pass functions that capture no environment at all. Understanding the difference between `fn` (a pointer to a function) and `Fn` / `FnMut` / `FnOnce` (closure traits) unlocks a powerful set of patterns for higher-order programming.

## Function Pointers: the `fn` Type

The `fn` type (lowercase) is a *function pointer*—a value that holds the address of a specific function. It is *not* a trait; it is a concrete type. A variable of type `fn(i32) -> i32` holds the address of any function that accepts an `i32` and returns an `i32`.

```rust
fn double(x: i32) -> i32 {
    x * 2
}

fn apply(f: fn(i32) -> i32, value: i32) -> i32 {
    f(value)
}

fn main() {
    // Pass a named function as a function pointer
    let result = apply(double, 5);
    println!("{}", result); // 10

    // Store a function pointer in a variable
    let op: fn(i32) -> i32 = double;
    println!("{}", op(7)); // 14
}
```

### `fn` vs `Fn`

| Feature | `fn` pointer | `Fn` / `FnMut` / `FnOnce` traits |
|---|---|---|
| Captures environment? | ❌ No | ✅ Yes (closures) |
| Concrete type? | ✅ Yes | ❌ No (trait) |
| Stored without `Box`? | ✅ Yes | ✅ (if sized) |
| Works in `extern "C"`? | ✅ Yes | ❌ No |

Because `fn` pointers implement all three closure traits (`Fn`, `FnMut`, and `FnOnce`), a function that accepts a `Fn(i32) -> i32` bound will also accept a plain `fn(i32) -> i32` pointer:

```rust
fn call_with_five<F: Fn(i32) -> i32>(f: F) -> i32 {
    f(5)
}

fn triple(x: i32) -> i32 { x * 3 }

fn main() {
    // Works with a closure
    let result1 = call_with_five(|x| x + 1);
    println!("{}", result1); // 6

    // Also works with a function pointer
    let result2 = call_with_five(triple);
    println!("{}", result2); // 15
}
```

## Passing Functions to Standard Library Methods

Function pointers are especially useful with iterator adapter methods like `map`, `filter`, and `for_each`. Passing a named function (or even a *type constructor*) instead of a closure can make code more concise:

```rust
fn is_even(n: &i32) -> bool {
    n % 2 == 0
}

fn square(n: i32) -> i32 {
    n * n
}

fn main() {
    let numbers = vec![1, 2, 3, 4, 5, 6];

    // Using named functions instead of closures
    let evens: Vec<i32> = numbers.iter().copied().filter(is_even).collect();
    let squares: Vec<i32> = numbers.iter().copied().map(square).collect();

    println!("Evens:   {:?}", evens);   // [2, 4, 6]
    println!("Squares: {:?}", squares); // [1, 4, 9, 16, 25, 36]
}
```

### Enum Variants as Function Pointers

Tuple-struct enum variants are, behind the scenes, initialiser functions. This means you can use them directly as function pointers:

```rust
#[derive(Debug)]
enum Status {
    Value(u32),
    Error(String),
}

fn main() {
    // Status::Value is fn(u32) -> Status — use it as a function pointer!
    let successes: Vec<Status> = (0u32..5).map(Status::Value).collect();
    println!("{:?}", successes);
    // [Value(0), Value(1), Value(2), Value(3), Value(4)]

    // Same trick with tuple structs
    #[derive(Debug)]
    struct Wrapper(i32);
    let wrapped: Vec<Wrapper> = vec![1, 2, 3].into_iter().map(Wrapper).collect();
    println!("{:?}", wrapped); // [Wrapper(1), Wrapper(2), Wrapper(3)]
}
```

## Returning Closures from Functions

Unlike function pointers, closures have anonymous types generated by the compiler. You cannot name the type in a return position, so you have two options:

### Option 1: `Box<dyn Fn(...)>` — Heap-allocated, dynamic dispatch

```rust
fn make_adder(x: i32) -> Box<dyn Fn(i32) -> i32> {
    Box::new(move |y| x + y)   // move captures x by value
}

fn main() {
    let add5 = make_adder(5);
    let add10 = make_adder(10);

    println!("{}", add5(3));   // 8
    println!("{}", add10(3));  // 13

    // Store a collection of closures — only possible with Box<dyn Fn>
    let ops: Vec<Box<dyn Fn(i32) -> i32>> = vec![
        make_adder(1),
        make_adder(100),
        Box::new(|x| x * 2),
    ];
    for op in &ops {
        println!("{}", op(10));
    }
    // 11, 110, 20
}
```

### Option 2: `impl Fn(...)` — Static dispatch, zero overhead (Rust 1.26+)

```rust
fn make_multiplier(factor: i32) -> impl Fn(i32) -> i32 {
    move |x| x * factor
}

fn main() {
    let triple = make_multiplier(3);
    println!("{}", triple(7));  // 21
}
```

`impl Fn` is monomorphised at compile time (zero-cost), but every call site produces a *different* concrete type, so you cannot store `impl Fn` closures in a `Vec` heterogeneously—use `Box<dyn Fn>` for that.

## Returning Different Closure Types Conditionally

When a function conditionally returns one of two different closures, even if they have the same signature the compiler considers them different types. The fix is `Box<dyn Fn(...)>`:

```rust
fn make_transform(double: bool) -> Box<dyn Fn(i32) -> i32> {
    if double {
        Box::new(|x| x * 2)
    } else {
        Box::new(|x| x + 10)
    }
}

fn main() {
    let t1 = make_transform(true);
    let t2 = make_transform(false);
    println!("{}", t1(5));  // 10
    println!("{}", t2(5));  // 15
}
```

## Function Pointers in FFI

When interacting with C via FFI, you sometimes need to pass a callback. C callbacks are bare function pointers with no environment, so you must use `fn` (not `Box<dyn Fn>`) and annotate with the correct calling convention:

```rust
// A type alias for a C-compatible callback
type CCallback = extern "C" fn(i32) -> i32;

extern "C" fn my_callback(x: i32) -> i32 {
    x * x
}

// Hypothetical FFI function that takes a callback
extern "C" {
    fn register_callback(cb: CCallback);
}

fn main() {
    unsafe {
        register_callback(my_callback);
    }
}
```

Using `Box<dyn Fn>` here would not work because it is a fat pointer (two words) and includes a vtable—not a plain C function pointer. Only bare `fn` pointers are ABI-compatible with C callbacks.

---

## Summary

- `fn` is a *type* (a function pointer); `Fn`, `FnMut`, and `FnOnce` are *traits* (implemented by closures and `fn` pointers alike).
- Function pointers implement all three `Fn` traits, so they can be passed wherever a closure bound is required.
- To return a closure from a function, use `impl Fn(...)` for zero-cost static dispatch or `Box<dyn Fn(...)>` for dynamic dispatch and heterogeneous collections.
- In FFI contexts, only bare `fn` pointers work as C callbacks.
